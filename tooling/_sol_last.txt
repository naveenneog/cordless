## Product decision

Your interpretation is right, with one correction: a CLI cannot realistically become Windows Terminal/Ghostty window chrome without becoming a GUI terminal emulator. Borrow their **home, switcher, restore, and command workflows**, not their renderer, tabs, or panes.

Core product:

```text
cordless CLI/TUI + persistent daemon + mobile/web client
```

Electron becomes optional and receives no v0.6 investment.

## Q1 — `cordless` starting screen

### (a) Thin TUI client of a persistent daemon

**Do not run the daemon in-process with the dashboard.**

`cordless` with no arguments:

1. Query the daemon through its authenticated loopback endpoint.
2. If unavailable, invoke the registered daemon start operation.
3. Wait for health for up to approximately 10 seconds.
4. Enter the dashboard.
5. `q` or Ctrl-C exits only the dashboard. PTYs and phone connections continue.

The daemon remains:

- a separate process;
- registered at login by `cordless install`;
- protected by the existing exclusive lock;
- owner of PTYs, pairing state, devices, and restore manifest.

This gives predictable ownership and means a dashboard crash or terminal close cannot kill sessions.

If the daemon is not installed, `cordless` may start it detached for the current login, but show:

```text
Daemon started temporarily. Run `cordless install` to start it automatically at login.
```

Never repeatedly spawn blindly. Query status, acquire the same lock, and treat “already running” as success.

### (b) v0.6 dashboard scope

Build a focused full-screen TUI, not a pane framework:

```text
┌ cordless ──────────────────────────────────────────────┐
│ Daemon  ● running    v0.6.0    Tailscale ● connected  │
│ URL     http://100.90.12.34:7443                      │
├ Pair a phone ───────────────┬ Sessions ────────────────┤
│                             │ > Shell   ~/src/app      │
│        QR CODE              │   Claude  ~/src/api      │
│                             │   Codex   ~/src/site     │
│ Expires in 04:31            │                          │
├─────────────────────────────┴──────────────────────────┤
│ Enter attach  n new  p new QR  d devices  r refresh    │
│ x kill        q quit dashboard                         │
└─────────────────────────────────────────────────────────┘
```

Hotkeys:

- `↑/↓` or `j/k`: select session
- `Enter`/`o`: attach
- `n`: create session
- `p`: replace pairing QR
- `d`: device/revocation view
- `x`: kill after confirmation
- `r`: refresh
- `q`/Ctrl-C: leave dashboard, not daemon

Implementation should use the alternate screen, raw keyboard mode, ANSI cursor positioning, and one render loop. Avoid React/Ink and a large TUI framework for this fixed layout. Restore cursor visibility, raw mode, and alternate-screen state in `finally`, on signals, and on uncaught errors.

Handle small terminals:

- Below the QR’s required dimensions, hide the QR.
- Show the pairing URL and: `Enlarge terminal or press p to retry`.
- Never truncate the secret-bearing URL into an unusable value.
- Do not write pairing URLs to logs.

### (c) Include minimal interactive session attach

Because the owner said **CLI first**, a dashboard that can only inspect sessions would feel incomplete. Include a minimal tmux-like attach mode, but not embedded panes.

On `Enter`:

1. Leave or suspend the dashboard alternate screen.
2. Attach through the existing authenticated local WebSocket protocol.
3. Put stdin into raw mode.
4. Write replay/live PTY output directly to stdout.
5. Forward stdin bytes to `session.input`.
6. Forward terminal resize events to `session.resize`.
7. Use `Ctrl-]`, then `d` as the detach chord.
8. Restore raw mode and redraw the dashboard.

Ctrl-C must be forwarded to the PTY while attached; it must not terminate cordless.

The host terminal remains the terminal renderer. You are **not** writing another xterm.js or ConPTY renderer.

Important implementation details:

- Reset the host terminal before applying a reset snapshot.
- Preserve output byte ordering.
- Respect `stdout.write()` backpressure.
- Advance/ACK sequence only after the corresponding write callback.
- On disconnect, leave attach mode cleanly and return to the dashboard.
- Always restore stdin raw mode and cursor state in `finally`.
- Test Windows Terminal, legacy `conhost`, PowerShell, cmd, and SSH terminals.
- Do not promise that arbitrary full-screen TUIs will survive simultaneous phone/local clients with conflicting resizes; last resize still wins.

Also expose the same behavior directly:

```bash
cordless attach <session-id-or-prefix>
```

## Q2 — Feature cut and ranking

### Ship in v0.6

1. **Welcome/home dashboard with QR**
2. **Session list and selector**
3. **Minimal interactive attach/detach**
4. **Create, kill, and device revoke actions**
5. **Existing session restore/reopen**
6. **Basic color/no-color support**
   - Respect `NO_COLOR`.
   - Do not build a theme editor.

### Defer

7. **Command palette**
   - Hotkeys and CLI commands cover v0.6.
8. **Workspaces**
   - The restore manifest already reopens sessions; named workspace composition can wait.
9. **Tabs**
   - The dashboard session switcher is enough.
   - Do not compete with Windows Terminal/iTerm tabs.
10. **Panes/splits**
    - Considerably more input routing, resize, rendering, and focus state.
11. **Themes**
    - Cosmetic after packaging and attach reliability.

“Warp blocks” are not a v0.6 feature. They require shell integration and command-boundary tracking, not merely TUI styling.

## Q3 — No-Node packaging

### Decision: Node SEA executable plus external native payload

Use **Node SEA**, not `pkg`, for the primary Windows executable.

Distribution:

```text
Cordless/
  cordless.exe
  resources/
    node_modules/
      node-pty/
        package.json
        lib/...
        build/Release/*.node
```

This gives:

- no system Node/npm prerequisite;
- a real `cordless.exe`;
- one pinned Node runtime/ABI;
- straightforward Authenticode signing;
- no dependency on `pkg`’s virtual filesystem patches.

It is not literally one file because native addons must exist as real filesystem files. Market it as a **self-contained installation**, not a single-file binary.

### Build pattern

1. Bundle the application into one CommonJS SEA entry using esbuild.
2. Exclude `node-pty` from that bundle.
3. Generate the SEA blob with the exact pinned Node release.
4. Inject the blob into a copy of that same Node executable.
5. Install a platform/architecture-specific `node-pty` beside it.
6. Load external `node-pty` using `createRequire()` rooted in the installed resources directory.

Conceptually:

```js
import { createRequire } from "node:module";
import path from "node:path";

const resources = path.join(path.dirname(process.execPath), "resources");
const externalRequire = createRequire(
  path.join(resources, "node_modules", "_cordless-loader.cjs"),
);

const pty = externalRequire("node-pty");
```

Do not depend on `process.cwd()`. Services and scheduled tasks start with unpredictable working directories.

### Native-module traps

- `pty.node` cannot be loaded directly from an embedded SEA asset or virtual filesystem.
- The addon ABI must exactly match the Node version used to create `cordless.exe`.
- Build/install `node-pty` independently for each OS and architecture.
- Do not copy Windows `node_modules` to macOS/Linux.
- Preserve node-pty’s expected package layout; avoid a custom patched path unless necessary.
- Keep native files in a stable install directory during upgrades.
- Sign only after SEA injection.
- The Windows Task Scheduler action should run:
  ```text
  "...\cordless.exe" start --foreground
  ```
- Ensure service mode is windowless. If the SEA executable inherits Node’s console subsystem, the scheduled task should run hidden; a future tiny GUI-subsystem launcher is optional.

### Why not the alternatives?

- **`@yao-pkg/pkg`**: workable, but native-addon extraction and virtual filesystem behavior add another packaging-specific abstraction.
- **SEA with embedded `pty.node` extraction**: possible but unnecessarily brittle; extraction, versioned cache, locking, antivirus, and integrity verification become your problem.
- **Private Node runtime plus JS files**: most boring fallback if SEA blocks development, but it produces a directory-level runtime and less polished command entry point.

### Required packaged tests

Test the installed artifact, not only repository execution:

```text
cordless --version
cordless start
cordless status
cordless
cordless pair
cordless attach <id>
cordless stop
cordless install
reboot/login autostart
```

Automate an isolated Windows VM test that:

- has no system Node;
- starts a real `pwsh.exe` PTY;
- exchanges input/output;
- reconnects and replays output;
- verifies Task Scheduler startup;
- verifies uninstall leaves or purges data according to flags.

## Q4 — Command surface and pairing ownership

Keep:

```text
cordless                     # dashboard
cordless start               # detached by default
cordless start --foreground  # service/debug mode
cordless stop
cordless status
cordless install
cordless uninstall
cordless pair
cordless devices
cordless attach <id>
cordless sessions
```

Useful cleanup:

```text
cordless new [shell|claude|codex] [--cwd ...] [--title ...]
cordless kill <id>
cordless logs
cordless doctor
```

`doctor` should check:

- daemon and protocol versions;
- port/listener;
- Tailscale installation/address;
- firewall reachability/configuration;
- configured shell/profile executables;
- data-directory permissions;
- autostart registration.

### Pairing state must remain daemon-owned

Neither the dashboard nor `cordless pair` should directly mutate a pending-pair file or in-memory map.

Both call the same authenticated local daemon operation:

```json
{
  "type": "pairing.create",
  "requestId": "..."
}
```

Requirements:

- Caller uses the loopback-scoped local credential.
- Actual socket peer must be loopback.
- Secret is 256 random bits, single-use, five-minute TTL.
- Replacing a dashboard QR invalidates that dashboard’s previous secret.
- `cordless pair` gets its own issuer/request ID and may coexist with the dashboard QR.
- Limit active pending pairs globally, e.g. three.
- Rate-limit creation and redemption separately.
- Dashboard sends `pairing.cancel` on clean exit.
- Daemon expiry remains authoritative when the dashboard crashes.

Do not let `cordless pair` instantiate a second private pairing store. The running daemon must be the only process capable of minting redeemable secrets.

The dashboard can refresh an expired QR automatically, but avoid rotating a still-valid QR every few seconds. Use:

- five-minute validity;
- visible countdown;
- automatic replacement only after expiry;
- `p` for manual replacement.

Prefer Tailscale in the encoded URL, then an explicitly enabled LAN address. Never encode loopback for phone pairing.

## Q5 — v0.6 cut line

### Ships

- Windows-first self-contained installer with no system Node requirement.
- Node SEA `cordless.exe` plus packaged Node-ABI `node-pty`.
- Existing persistent daemon and login autostart.
- `cordless` full-screen dashboard.
- Daemon/Tailscale/reachability status.
- Live single-use pairing QR with countdown and regeneration.
- Session list, create, kill, and refresh.
- Device list and revocation.
- Minimal local interactive attach/detach.
- `cordless attach`, `new`, `sessions`, and `doctor`.
- Existing mobile APK and web client.
- Existing replay, snapshot, authentication, restore, CORS, CSP, and rate-limit behavior.
- Migration that detects and reuses the existing `~/.cordless` data.
- Unsigned Windows build clearly labeled as such.
- CI test of the installed artifact on Windows without system Node.

### Defers

- Electron as the default product; retain it as experimental/secondary.
- TUI tabs and panes.
- Command palette.
- Warp-style blocks or shell integration.
- Named workspaces/layouts.
- Rich theme system.
- Literal one-file packaging of `node-pty`.
- macOS/Linux polished installers and service integration.
- Auto-update.
- Azure Trusted Signing, until the owner completes account and identity setup.
- Automatic broad LAN exposure.

The v0.6 product statement should be:

> Install cordless, run `cordless`, scan the pairing QR, and manage or attach to persistent coding sessions from the same terminal or your phone.
